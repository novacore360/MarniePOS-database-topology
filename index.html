How each pipeline protects itself

Every run is cheap by default and defensive by design. Expand a section for the mechanics.

Free-tier hosts idle down between requests, and an idle host can't fire its own cron job at exactly 05:00 or 17:00. To cover that gap, every incoming request — a health check, a status call, or a real cron tick — re-checks whether the most recent scheduled slot actually ran.
  • If the check happens inside the grace window (default 180 minutes) and that slot hasn't run yet, it runs now, labeled as a catch-up.
  • A small marker document, __backup_metadata__/__global__, records the last completed slot so the same window is never repeated.
  • Outside the grace window, the service does nothing and waits for the next real slot — this is a safety net, not a substitute for a host that never sleeps.

The 6 PM slot is allowed to fail and try again, within limits tracked in a backup_slots table.
  • Up to 5 attempts inside a rolling one-hour window before backing off.
  • If a window is exhausted, a fresh hourly window opens anchored to the first attempt — 6:04 PM, then 7:04 PM, and so on.
  • Retries stop 5 hours after the original slot — i.e. no more retries after 11 PM. The slot is then marked abandoned until the next day's 6 PM slot.
The live numbers on the status card above always reflect whatever the /admin endpoint currently reports, so if this value is tuned again later, the page follows it automatically.

Both pipelines are time-gated and diff-aware, so an ordinary day is close to free.
  • Outside the scheduled hour, neither service reads from Firestore at all.
  • Each collection keeps a checksum map from its last sync; only changed, new, or removed documents are written.
  • A collection that reads back empty is treated as suspicious, not authoritative — the backup is skipped rather than used to wipe good data.
  • Writes are batched — 400 documents per batch on the Firestore mirror, 400 rows per statement on Postgres.
